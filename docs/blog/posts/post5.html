Debugging Specificity

Debugging Specificity Conflicts in Real Projects

Published: November 25, 2025

“Why isn’t my CSS working?” — The #1 question every front-end developer asks weekly. 95 % of the time: specificity.

Step-by-Step Debugging Process

  1. Open DevTools → Elements panel
  2. Select the element that’s misbehaving
  3. Look at the Styles pane — crossed-out rules = lost to specificity
  4. Copy the winning and losing selectors
  5. Paste them here → see exactly who wins and why

Common Culprits

  • Bootstrap/Tailwind utility classes (high specificity)
  • Deep nesting in CSS modules or SCSS
  • Third-party libraries using #ids
  • Legacy CSS with long selector chains
Pro tip: Use this tool during code reviews — paste component selectors and catch conflicts before they ship.

FAQ

How do I override a library’s styles?

Match or increase specificity — or use :where() for low-specificity base styles.

Why does my hover state not work?

Often because a non-hover rule has higher specificity.
